# Orbix Lavalink playback sessions and interactive queue controls

This change adds `play`, `skip`, `pause`, `resume`, `join`, `dc`, and `queue` around per-guild Shoukaku players, with queued, now-playing, and interactive queue Components V2 surfaces. Active-session mutations are serialized and player events feed a persistent panel, but lifecycle and transition gaps can leave commands hung, guilds unable to rejoin, or Orbix out of sync with Lavalink. The most serious defects are seconds/milliseconds confusion in Shoukaku options, incomplete Shoukaku cleanup after Discord disconnects, and failed tracks being recycled by queue loop. The configured public node is also a third-party trust boundary, while malformed TLS configuration fails open.

**Watch for:** **confirmed** multi-hour Shoukaku timeouts; **confirmed** Discord-side disconnects that block later joins; **confirmed** infinite retry of failed/stuck tracks in queue-loop mode; **confirmed** disconnect/reconnect teardown races; **confirmed** old player panels controlling later sessions.

## High-level view

**Confirmed:** Shoukaku v4.3 interprets three configured timeouts as seconds, not milliseconds. The pinned 4.3.0 implementation also fails recovery after an initial connection attempt fails, while Orbix's terminal-node handler leaves affected sessions active and misreads the event's count argument.

**Confirmed:** Discord-side voice removal and guild deletion remove only Orbix's session, leaving Shoukaku's connection/player records behind. Normal disconnect exposes the guild before asynchronous Shoukaku teardown completes, allowing a concurrent join to collide with or be removed by old cleanup.

**Confirmed:** Queue loop requeues `loadFailed` and `stuck` tracks indefinitely. Play-now, previous, and track-loop replay commit queue/history changes before replacement succeeds, so REST failure can strand waiting tracks or desynchronize local and remote playback.

**Confirmed:** Player IDs identify only a guild, old panels are not retired when the panel channel changes, and toggle/volume decisions occur outside the operation queue. **Likely:** destructive controls and player events can target the wrong track around an end transition, and uncached member lookup before acknowledgement can miss Discord's three-second deadline.

**Confirmed:** The automatic public Lavalink fallback receives searches and Discord voice-session material. An invalid `LAVALINK_SECURE` value silently selects plaintext instead of rejecting the configuration.

**Confirmed:** The first single track is reported at queue position zero, the lockfile also upgrades Discord runtime packages, and no repository test exercises music lifecycle or interactions.

<summary>Details</summary>

## Shoukaku's seconds-based options produce multi-hour failure windows

**Confirmed:** `src/utils/music.js:121-128` sets `reconnectInterval: 5_000`, `restTimeout: 15_000`, and `voiceConnectionTimeout: 15_000`; Shoukaku 4.3 multiplies each by 1,000 internally. Reconnects wait 83 minutes, while REST and voice handshakes can hold a command and its session queue for 4 hours 10 minutes. The reconnect log at `src/utils/music.js:150-152` also labels this seconds value as milliseconds. Use `5`, `15`, and `15`, and protect the units with an option-contract test.

## Pinned Shoukaku 4.3.0 drops a recovered connection

**Confirmed:** the dependency pinned at `package.json:22` retains `connectError` after any failed WebSocket attempt in its installed `Node.connect()`. A later successful attempt still enters the failure branch, closes the new socket, emits `disconnect`, and removes the node; only an immediate first-attempt success establishes a node. Upgrade to a fixed release or carry a reviewed patch, then test a fail-once-then-connect sequence.

## Terminal Lavalink loss leaves live but unusable sessions

**Confirmed:** `src/utils/music.js:146-148` only logs Shoukaku's terminal `disconnect`. With one node and `moveOnDisconnect: false`, Shoukaku removes the node but retains guild connections/players; Orbix keeps `session.current`, receives no later end event, and never starts idle cleanup. Searches reject, controls target a stale player, and the bot can remain in voice indefinitely. The callback's `count` is moved-player count—not reconnect attempts—and is always zero here. Close affected sessions or implement tested node recreation/resume, and correct the event telemetry.

## Discord-side disconnects leave Shoukaku's guild slot occupied

**Confirmed:** guild deletion and bot voice removal pass `leaveVoice: false` at `src/utils/music.js:158-162` and `src/utils/music.js:885-889`. Orbix deletes its session, but Shoukaku's raw voice update only marks its connection disconnected; the connection/player maps remain, so every later join throws `This guild already have an existing connection` until restart. Always run Shoukaku cleanup on these paths; its gateway leave is already a no-op once disconnected, but its maps and Lavalink player still need removal.

## Session deletion races new joins against old teardown

**Confirmed:** teardown removes the session at `src/utils/music.js:853-855` before waiting for operations and `leaveVoiceChannel` at `src/utils/music.js:860-864`. A new join can see no Orbix session while Shoukaku still has the old connection, or start after connection removal while old player destruction is pending; that old delete/map cleanup can then race the replacement player. Keep a per-guild lifecycle tombstone until teardown finishes and serialize connect, disconnect, and initial session creation under one guild lock.

## Queue loop turns failed tracks into poison entries

**Confirmed:** `loadFailed` and `stuck` both reach `advanceUnlocked` (`src/utils/music.js:364-372`, `src/utils/music.js:397-408`), which appends the previous track whenever queue loop is active regardless of reason (`src/utils/music.js:627-630`). A one-track queue retries and reports the same failure forever; a larger queue retries it every cycle. Exclude failed/stuck tracks from loop recycling and add an attempt cap.

## Direct replacement failure strands queue and history state

**Confirmed:** track-loop replay, previous, and play-now (`src/utils/music.js:621-624`, `src/utils/music.js:650-672`, `src/utils/music.js:740-758`) mutate state before `playEntryUnlocked` succeeds. That method clears `session.current` on REST rejection, but these callers do not restore removed state, advance through the normal fallback, or schedule idle cleanup. Waiting tracks can remain permanently idle, while a timed-out request may have succeeded remotely. Make replacement transactional around an expected current entry and reconcile or restore before normal queue fallback.

## Guild-only player IDs leave old panels authoritative

**Confirmed:** player IDs contain only `action:guildId` at `src/utils/musicUi.js:120-125`; handlers check neither session generation nor authoritative message ID. `textChannelId` can move through connect, enqueue, or queue, and `upsertPlayerPanel` replaces its message pointer without retiring the prior panel (`src/utils/musicUi.js:557-578`). That panel can later pause, skip, or stop unrelated playback, including a new session. Include a session nonce/current-entry ID, require the active panel message, and deactivate a panel before moving ownership.

## End-boundary actions lack a track-generation precondition

**Likely:** `skip()` acts on whichever entry is current when its serialized callback runs (`src/utils/music.js:645-647`), not the entry shown at invocation, while start/end/stuck events are correlated only by encoded track (`src/utils/music.js:335-360`, `src/utils/music.js:397-400`). A natural end immediately before a queued skip can consume the next track; duplicate encodings make a late event indistinguishable from the replacement. Pass expected queue-entry IDs into destructive actions and correlate events with the `queueEntryId` already sent in track `userData`, or another playback generation.

## Toggle and volume decisions happen outside serialization

**Confirmed:** `src/utils/musicUi.js:461-481` reads `session.paused` and computes `session.volume ± 10` before calling serialized manager methods. Concurrent clicks can both choose pause or submit the same target volume, collapsing two intended actions into one. Implement atomic `togglePause()` and `adjustVolume(delta)` operations inside `MusicManager.serialize`.

## Member lookup precedes interaction acknowledgement

**Likely:** both control handlers may await `guild.members.fetch()` at `src/utils/musicUi.js:410-440` before `reply`, `update`, or `deferUpdate`. Slow REST can cross Discord's three-second deadline, and the failed response is swallowed. Acknowledge first, then validate and edit the deferred response or send an ephemeral follow-up.

## First playback is reported at queue position zero

**Confirmed:** `enqueue()` returns zero when no track is active (`src/utils/music.js:519`), and the single-track response renders `Queued at position 0` (`src/utils/musicUi.js:135-137`) even though that entry starts immediately. Return a now-playing status for the first entry or use one-based positions consistently.

## Public Lavalink is an automatic credential trust boundary

**Confirmed:** `.env.example:11-16` and `src/config.js:21-26` fall back automatically to a shared public node. By protocol, that operator receives music queries plus the ephemeral Discord voice token, endpoint, and session ID. Make public-node use opt-in, document the trust decision, and prefer an operator-controlled production node.

## Invalid secure-mode values silently disable TLS

**Confirmed:** `parseBoolean` maps every nonempty value outside its true allow-list to false (`src/config.js:36-41`). A typo such as `LAVALINK_SECURE=treu` disables TLS, potentially exposing node authentication, searches, and voice-session material on a plaintext-capable endpoint. Parse explicit true and false forms and reject everything else so transport security fails closed.

## Lockfile installation also upgrades the Discord runtime

**Confirmed:** `package-lock.json` moves `discord.js` 14.26.4 to 14.27.0, `@discordjs/rest` 2.6.1 to 2.6.2, and related packages although the Discord manifest line is unchanged. This affects every command and interaction. Revert unrelated lock movement or separate and test the Discord upgrade intentionally.

## Music behavior has no regression coverage

**Confirmed:** all nine existing tests pass, but no test references music, Shoukaku, Lavalink, or the new custom-ID prefixes. Lifecycle, event ordering, queue failures, stale controls, acknowledgement, and Components V2 limits are unprotected. Add fake-player manager tests and interaction/payload tests covering these boundaries.
